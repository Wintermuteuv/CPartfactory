# Cursed Pit — Topology Decision

**Статус:** Decision point для передачи в code-Claude (ADR-010, ADR-011)
**Контекст:** пересмотр Phase 1 генератора уровней после выявления, что DownwardDFS описывает только один частный случай топологии и неприменим к Dwarven Origin без семантического подлога.

---

## 1. Топология — family of generators

**Подход A с минимальным сабсетом на MVP.**

Phase 1 становится семейством генераторов. Диспатч по `LevelProfile.Origin` через фабрику.

**Две ветви на MVP:**

| Generator | Используется для |
|-----------|------------------|
| `URoomsAndCorridorsGenerator` (BSP-стиль) | `DwarvenTech`, `DwarvenResidential`, `HumanAdapted` |
| `UDownwardDfsGenerator` (текущий) | `Natural`, `HumanFrontier` |

**Принципы реализации BSP-ветви:**
- Намеренно тупая на MVP: прямоугольная разбивка grid → коридоры по центрам граней соседних прямоугольников.
- Без вариативных размеров комнат, без многоуровневой иерархии, без поворотов коридоров.
- Этого достаточно, чтобы получить читаемое "это гномий этаж" вместо "это лабиринт с гномьими обоями".

**Обоснование выбора A, а не C (BSP pre-pass перед DFS):**
- C-слабый вариант (BSP размечает комнаты + DFS заполняет остальное) производит третью топологию — не гномью и не природную. Лорно не осмысленна.
- C-сильный вариант (BSP только для Dwarven*, DFS не запускается) — это де-факто A с одной реализованной веткой. Промежуточный шаг C — это код, который потом выкинется.
- Обе ветви будут в финальной игре в любом случае. Делать их сразу — экономия.

## 2. FloorZ — per volume, не per cell

**Правило:** FloorZ — свойство логической единицы пространства, не cell.

| Origin | Что считается логической единицей |
|--------|-----------------------------------|
| Dwarven* | "Комната" (volume из BSP-разбиения) |
| Natural | "Цельная пещера" (volume из группировки cells одного SpaceType) |
| Любой | "Коридор" — отдельный тонкий volume |

**Новая фаза между Phase 1 и AssignZ:**

```
Phase 1:    Generator → cells + connections (топология)
Phase 1.5:  VolumeBuilder → logical volumes
              - Dwarven*: volumes из BSP-алгоритма
              - Natural:  группировка cells одного SpaceType=Cavern,
                          Corridor-cells = отдельные тонкие volumes
Phase 2:    Populate → SpaceType, Material per volume
Phase 2.5:  AssignZ → FloorZ per volume
              - Dwarven volume → constant FloorZ
              - Natural volume → noise field, FloorZ варьируется внутри
              - Corridor между разными FloorZ → slope или ступени
```

**Что это даёт:**
- Гномьи комнаты получают ровный пол внутри (как и должно быть).
- Природные пещеры сохраняют органическую вариативность.
- Slope становится производной границ между volumes, а не самостоятельной логикой.

---

## 3. Slope (#5) — отложить

Геометрический slope нельзя решать в отрыве от Origin и до появления VolumeBuilder.

**Причина:** склоны — это переходы между volumes с разным FloorZ. Без понятия volume нет правильного места для slope-логики.

**После VolumeBuilder:** склоны появляются естественно на границах volumes с разными FloorZ. Это **производная**, не отдельная фича.

## 4. AuthoredLocation — ADR-006 → ADR-011

Текущая бинарная модель `EDungeonLocationKind { Procedural, Authored }` **не покрывает реальный production workflow.**

**Реальный спектр:**

| Use-case | Что делает генератор | Что делает дизайнер вручную |
|----------|---------------------|----------------------------|
| Обычный этаж | Всё | Ничего |
| Темница (-1) | Сгенерил один раз | Декорировал → экспорт в `.umap` |
| Финальный этаж (-60) | Большую часть | Врезает 1-2 ключевые комнаты (финальная дверь) |
| Pure authored (если когда-то понадобится) | Ничего | Всё с нуля |

**Решение:** переход от enum к двум ортогональным флагам на `LevelProfile`:

```cpp
bool bRuntimeGenerated;             // false = карта baked в .umap
TArray<FAuthoredPatch> AuthoredPatches;  // комнаты, врезаемые поверх процедуры
```

**Маппинг use-case → флаги:**

| Use-case | bRuntimeGenerated | AuthoredPatches |
|----------|-------------------|-----------------|
| Обычный этаж | true | пусто |
| Темница (-1) | false | пусто |
| Финальный этаж (-60) | true | финальная дверь + подходы |
| Pure authored | false | пусто |

**Почему так:**
- Два флага точнее описывают реальные комбинации, чем enum из 3-4 значений.
- Workflow темницы — это `Procedural Snapshot`: сгенерили → декорировали в редакторе → сохранили как `.umap`. Это не "pure authored", это "baked procedural".
- Workflow -60 — это `Hybrid`: runtime procedural + врезанные авторские патчи.

## 5. Tactical cleanup — делать параллельно

Пункты **#1, #3, #4, #6** из текущего backlog — закрывать сейчас, не дожидаясь топологического решения.

**Почему безопасно:**
- **#1 (passability)** — бинарное "проходимо/нет" по dZ. Живёт в любой топологии.
- **#3 (cleanup)** — sparsity-дыры закрывать стенами в любом случае.
- **#4 (encounter density)** — это Phase 3 (AssignOccupants), работает на готовом графе, не знает откуда он.
- **#6 (удаление UDfsRefiner)** — уборка мусора, в новой архитектуре его нет.

**Что НЕ втягивать в cleanup:**
- Slope-логику в #1 (passability). Только бинарное "проходимо/нет" по порогу dZ.
- Любые попытки делать "комнаты" в текущем DFS post-hoc. Это работа для BSP-ветви.

## 6. Что фиксируется этим документом

1. Phase 1 — family of generators, диспатч по `Origin`.
2. На MVP — две ветви: `URoomsAndCorridorsGenerator` (BSP) и `UDownwardDfsGenerator` (DFS).
3. BSP-ветвь намеренно простая: прямоугольники + коридоры по центрам граней.
4. FloorZ работает per-volume через новую Phase 1.5 (VolumeBuilder).
5. Dwarven volumes → constant Z. Natural volumes → noise. Коридоры между разными Z → slope/ступени.
6. Slope (#5) откладывается до появления VolumeBuilder.
7. `EDungeonLocationKind` (enum) заменяется на два флага: `bRuntimeGenerated` + `AuthoredPatches`.
8. Cleanup #1+#3+#4+#6 делается параллельно с дизайн-сессией.

---

## 7. Что НЕ фиксируется

- Конкретные размеры прямоугольников BSP (минимум/максимум).
- Точный алгоритм группировки cells в Natural volumes.
- Конкретный noise function для FloorZ в Natural.
- Точные пороги dZ для passability в cleanup #1.
- Структура `FAuthoredPatch` (позиция, размер, способ интеграции с runtime-генерацией).

Это всё — материал для ADR-010 и ADR-011, которые пишет code-Claude в репо.

## 8. Открытые вопросы

1. **Ресурс на две ветви параллельно.** Реализация BSP-ветви — это +1 полноценный алгоритм. Если нагрузка ощущается высокой, возможен компромисс: начать с BSP-ветви, оставить DFS для авторских природных уровней позже.
2. **Стыковка volumes на границах cells.** Когда два volume встречаются по диагонали или через тонкий коридор — кто владеет промежуточным пространством? Edge cases для VolumeBuilder.
3. **AuthoredPatches и runtime-генерация.** Как именно патч "вырезает" место в процедурной карте — координатами cells, bounding box в мире, ссылкой на маркер? Решать в ADR-011.
4. **Миграция существующего кода.** Если что-то уже опирается на `EDungeonLocationKind` — план миграции в ADR-011.
