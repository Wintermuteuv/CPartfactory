## 0. Что это и зачем

**Art Factory** — локальный веб-инструмент для генерации визуальных референсов
(концепт-артов окружения) для игры Cursed Pit. Не для продакшен-ассетов, а для
**референсов**: насмотренность, мудборд, единый визуальный язык проекта.

**Ключевая ценность инструмента — консистентность стиля между артами**, а не
максимальное качество одиночной картинки. Это решение принято осознанно и
определяет всю архитектуру.

**Роль владельца проекта (Ivan):** направление и дизайн-интент. Весь код пишет
Claude. Выдавай готовые рабочие блоки, а не пошаговые ручные инструкции.

## 1. Решённые архитектурные вопросы (НЕ переоткрывать)

Эти решения уже приняты после анализа. Не предлагай альтернатив без явного запроса.

| Вопрос | Решение | Причина |
|--------|---------|---------|
| Провайдер генерации | Локальный **ComfyUI** (SDXL), гибрид с арендой GPU позже | Только локальный путь даёт консистентность как управляемый параметр |
| Базовая модель | **SDXL** (не Flux) | На 8 ГБ VRAM SDXL+IP-Adapter уживаются, Flux+IP-Adapter — нет |
| Механизм консистентности | **IP-Adapter** (style reference) + при необходимости **LoRA** | Фундамент единого визуального языка |
| Связь с ComfyUI | Локальный **HTTP API** ComfyUI (`127.0.0.1:8188`) | Node-бэкенд дёргает его программно |
| Источник промптов | Оси из `environment.md` (НЕ свободный текст диздока) | Формализованная вариативность вместо абстрактных слайдеров |
| Авто-цикл "анализ→re-анализ" | **ВЫКИНУТ из MVP** | Image-генераторы не управляются текстовой критикой; каждая правка = новый сэмпл. Заменён на batch→select→refine с фиксацией seed |

### Железо
- GPU: **RTX 5060 (8 ГБ VRAM)**, Blackwell sm_120 — требует PyTorch с CUDA 12.8+
- Установка ComfyUI: официальный portable, путь `D:\AI\ComfyUI_windows_portable\`
- Гибрид-плечо (аренда GPU для финальных Flux-артов) — **Этап 5, позже**, не MVP

### Ограничение 8 ГБ → стадийный процесс (важно для архитектуры)
Нельзя держать в VRAM всё разом (base + LoRA + IP-Adapter + ControlNet + upscaler).
Фабрика гоняет генерацию **стадиями**, не монолитным workflow:
1. **Черновой батч**: SDXL base + IP-Adapter, низкие шаги, малое разрешение, N вариантов
2. **Выбор**: пользователь тыкает лучший → фиксируется его seed
3. **Финал**: тот же seed на полных шагах + upscale (`4x-UltraSharp`) отдельным проходом

## 2. Визуальный стиль Cursed Pit (для базового промпта)

Направление: **мрачный low fantasy, industrial decay, cave horror, grounded
materials, ограниченная палитра**. Чем глубже — тем больше сюрреализма,
impossible geometry и визуальной нестабильности (это управляется параметром
глубины, см. ниже — НЕ хардкодить в базовый промпт).

Базовый стилевой текст применяется ко всем артам поверх осей. Точную формулировку
подбирать итеративно с эталонными изображениями через IP-Adapter.

## 3. Оси генерации (источник: environment.md)

Промпт собирается из комбинации осей, а НЕ из свободного текста. Это ядро PromptBuilder.

### 3.1 Material (5) — природа породы вокруг
`Stone` (базовый, нейтральный) · `Ice` (холодный, скользкий, отражающий) ·
`DwarvenMetal` (техногенный, обработанный) · `Organic` (мягкий, влажный, биоматериал) ·
`Crystal` (редкий, излучающий)

### 3.2 SpaceType (6) — функция и геометрия пространства
`Corridor` (узкий, плоский, линейный) · `Hall` (средний, регулярный, гномий стандарт) ·
`Shaft` (вертикальный) · `Cavern` (большой, неровный пол и потолок) ·
`Maze` (высокая плотность связей, дезориентация) · `Chasm` (провал, риск падения)

### 3.3 Origin (5) — кто построил пространство
`Natural` · `DwarvenTech` (функциональная инфраструктура) ·
`DwarvenResidential` (жилые/общественные) · `HumanAdapted` (надстройки поверх гномьего, только до -11) ·
`HumanFrontier` (заброшенные форпосты ниже зоны города)

### 3.4 Occupant (7 для MVP) — кто живёт сейчас
`None` · `Goblins` · `Ratfolk` · `HumanGarrison` (до -5) · `HumanExiles` ·
`DwarvenRemains` · `DeepHostile`
(НЕ MVP, только "История": `DwarvenLiving` и сюжетные NPC — не включать в первую версию)

### 3.5 Depth — глубина этажа (слайдер, диапазон -1 … -60)
Глубина — НЕ ось, а driver, который определяет три производных:

**A. AnomalyIntensity (степень сюрреализма/нестабильности):**
| Глубина | AnomalyIntensity |
|---------|------------------|
| -1 … -11 | 0 |
| -12 … -31 | 0.1 … 0.3 |
| -32 … -34 | 0.4 |
| -35 … -44 | 0.5 … 0.7 |
| -45 … -60 | 0.8 … 1.0 |

Чем выше — тем сильнее модификаторы промпта про impossible geometry, искажения
света/звука, мутации. Опционально: при высоком AnomalyIntensity снижать вес
IP-Adapter, чтобы картинка буквально "теряла стабильность" вглубь.

**B. ThermalZone (накладывает ЗАПРЕТЫ на Material/Occupant):**
- `Cooling` (-1…-31): запрещён Material `Organic`
- `DeepFreeze` (-32…-34): запрещены Material `Organic`, `Crystal`; разрешён в основном `Ice`/`Stone`
- `Reheating` (-35 и ниже): запрещён Material `Ice`; `Organic` разрешён только на положительных температурах

**C. Допустимые Origin/Occupant по глубине** (см. таблицу п.9 в environment.md;
кратко: человеческие фракции только в верхних этажах, DeepHostile глубже -10,
endgame -45…-60 = DwarvenResidential).

## 4. Валидация комбинаций (важно — иначе лорно-невозможные арты)

PromptBuilder должен **отклонять или предупреждать** о невозможных комбинациях,
используя три слоя правил из environment.md:
1. **Температурный запрет** (Material vs ThermalZone, см. 3.5.B)
2. **Глубинная раскладка** (Origin/Occupant vs Depth, таблица п.9)
3. **Origin-ограничения по глубине** (HumanAdapted только до -11, HumanGarrison до -5)

Например: `Organic + depth=-5` → невалидно (Organic запрещён в Cooling).
`HumanGarrison + depth=-30` → невалидно (гарнизон только до -5).

Реализация: загрузить правила как конфиг (можно отдельный `axis_rules.json`),
не хардкодить в коде — Ivan будет их корректировать.

## 5. Целевая архитектура

```
[Node backend]
  ├─ парсер environment.md (или статичный axes.json, выгруженный из него)
  ├─ axis_rules.json — правила валидации (температура, глубина, origin-лимиты)
  ├─ PromptBuilder: оси + depth → { базовый стиль + фразы осей + аномалия-модификаторы }
  │     └─ валидация комбинации перед сборкой
  ├─ ComfyUIClient: HTTP → 127.0.0.1:8188
  │     ├─ загрузка workflow-шаблона (JSON, экспортируется из ComfyUI в API-формате)
  │     ├─ подстановка промпта/seed/параметров в ноды workflow
  │     ├─ POST /prompt, опрос /history, забор результата
  │     └─ стадийность: draft-батч → финал по выбранному seed → upscale
  └─ хранилище:
        art_<material>_<space>_<origin>_<occupant>_d<depth>_seed<N>.png
        + рядом одноимённый .json (полный промпт, seed, все оси, версия стиля, дата)

[Web UI]
  ├─ дропдауны: Material / SpaceType / Origin / Occupant
  ├─ слайдер: Depth (-1…-60), показывает производные AnomalyIntensity/ThermalZone
  ├─ превью собранного промпта (РЕДАКТИРУЕМЫЙ перед отправкой)
  ├─ индикатор валидности комбинации (зелёный/предупреждение)
  ├─ кнопка "генерить N вариантов"
  └─ галерея результатов → выбор лучшего → "финализировать" (seed-lock + upscale)
```

### Технологии (предложение, обсуждаемо)
- Backend: Node + Express/Fastify
- Frontend: что-то лёгкое (можно vanilla + Vite, или React — на усмотрение)
- Без БД на старте: метаданные в JSON рядом с картинками; индекс — сканом папки

## 6. План разработки (порядок для Claude Code)

> Предусловие: ComfyUI уже установлен, запущен в API-режиме (`--listen` /
> доступен на `127.0.0.1:8188`), первая картинка через SDXL сгенерирована,
> IP-Adapter настроен с эталонными изображениями. Это делается ОТДЕЛЬНО (вне Code).

1. **Каркас проекта**: Node-бэкенд + пустой фронт,健康-чек связи с ComfyUI API.
2. **ComfyUIClient**: загрузка экспортированного workflow-JSON, подстановка
   параметров, отправка, опрос результата, сохранение PNG.
3. **axes.json + axis_rules.json**: выгрузить оси и правила из environment.md
   в конфиги. PromptBuilder + валидатор.
4. **Web UI**: дропдауны + слайдер depth + превью промпта + кнопка генерации.
5. **Батч + галерея**: N вариантов, сетка, выбор.
6. **Финализация**: seed-lock выбранного + upscale-проход + именование/метаданные.
7. (Позже) **Гибрид-плечо**: переключатель локально/аренда для Flux-финалов.

MVP = пункты 1–6. Авто-критику vision-моделью НЕ добавлять.

## 8. Чего НЕ делать

- НЕ добавлять авто-цикл "анализ→re-анализ" в MVP.
- НЕ переключать на Flux для локальной генерации (8 ГБ + IP-Adapter не тянет).
- НЕ хардкодить правила осей в код — выносить в JSON-конфиг.
- НЕ собирать промпт из свободного текста диздока — только из формализованных осей.
- НЕ строить монолитный workflow — соблюдать стадийность из-за VRAM.
- НЕ использовать localStorage/sessionStorage в UI-артефактах.
